# Анализ главных страниц аналогов (формы ввода)

## Критерии анализа (с позиции экспертной системы)

Опора на `AGENTS.md`: для продукционной ЭС нужны формализуемые факты, которые легко подключать в `TRAVEL_FACTS` и использовать в правилах (`TRAVEL_RULES`), с дальнейшим объяснением вывода.  
Опора на `docs/reports/materials`: важны цель и мотивация поездки, сезонность, тип потребителя, базовые потребности (передвижение/проживание/питание), социально-демографические и визовые факторы.

## Сводно по аналогам

| Сайт | Что можно ввести на главной | Типы данных | Польза для продукционной ЭС | Чего не хватает для ЭС |
|---|---|---|---|---|
| `ostrovok.html` | Направление (город/отель/аэропорт), заезд, выезд, состав гостей (напр. `2 гостей`), переключатели `Отдых/Работа`, `Только квартиры`; сервисно: email-подписка + согласие | Текст, даты, число/категория, булево, категориальный выбор | Хороший базовый набор фактов по размещению и цели поездки (частично покрывает «цель поездки», «проживание», «сезонность через даты») | Нет явного бюджета, вылета/транспорта, визовых ограничений, состава группы (дети/возраст), предпочтений по активности/питанию |
| `turscanner.html` | Город вылета (radio), месяц начала тура (множественный выбор + `Не важно`), `Без визы`; кнопка поиска | Категориальный выбор, множественный выбор, булево | Сильно выражены сезонность и визовый фактор; есть стартовый транспортный контекст (город вылета) | Нет поля цели поездки, направления как пользовательского ввода, бюджета как фильтра, длительности, состава туристов, требований к сервису |
| `tripster.ru` | Поиск направления: `Куда вы собираетесь?` (с подсказками по странам/городам); сервисно: email-подписка, логин (email/password) | Текстовый поиск, сервисные поля аккаунта | Быстрый сбор ключевого факта «куда» (направление/регион интереса) | Для консультационной ЭС почти нет туристских фактов: нет дат, бюджета, визы, состава группы, цели, ограничений и предпочтений |

## Краткие наблюдения по каждому аналогу

1. `Ostrovok` ближе всего к структуре фактов, пригодной для продукционных правил: уже есть цель (`Отдых/Работа`), даты, состав гостей и тип размещения.
2. `Turscanner` хорошо захватывает внешние ограничения (`город вылета`, `месяц`, `без визы`), но слабо описывает потребности конкретного туриста.
3. `Tripster` на главной странице решает задачу навигации по направлению, а не экспертной диагностики потребностей пользователя.

## Что важно взять в нашу форму (для `TRAVEL_FACTS`)

1. Цель/мотив поездки (отдых, познавательная, деловая, семейная и т.д.).
2. Сезонность и временные параметры (даты или месяц + гибкость дат).
3. География: откуда/куда (город вылета и направление).
4. Визовые ограничения.
5. Состав группы (взрослые/дети, семейный формат).
6. Бюджетный диапазон.
7. Длительность поездки.
8. Предпочтения по типу тура и сервису (активности, размещение, при необходимости питание).

## Итог

С точки зрения экспертной системы лучший ориентир среди текущих аналогов — `Ostrovok` (больше структурированных фактов на входе).  
Однако для требований проекта (масштаб правил, объяснимый вывод, ориентация на мотивы и тип туриста) нужно расширять форму за пределы аналогов: добавить мотив, бюджет, длительность, состав группы и ограничения, чтобы фактов хватало для плотной продукционной базы.
